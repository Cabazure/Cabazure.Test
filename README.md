## Overview

**Cabazure.Test** is a spiritual successor to [Atc.Test](https://github.com/atc-net/atc-test), rebuilt from the ground up for xUnit 3. It bundles xUnit 3, NSubstitute, AutoFixture, and FluentAssertions into a single package so you can focus on writing tests instead of wiring up infrastructure.

Interfaces and abstract classes are substituted by NSubstitute automatically. Parameters in theory tests are auto-generated by AutoFixture. Project-wide customizations register once via `[ModuleInitializer]` — no static constructors or manual bootstrapping required.

**Target framework:** `net9.0`

**Bundled packages (transitive — no extra references needed):**

| Package | Version |
|---|---|
| `xunit.v3` | 3.2.2 |
| `NSubstitute` | 5.3.0 |
| `AutoFixture` | 4.18.1 |
| `AutoFixture.AutoNSubstitute` | 4.18.1 |
| `FluentAssertions` | 7.0.0 |

```
dotnet add package Cabazure.Test
```

## Quick Start

```csharp
using Cabazure.Test.Attributes;
using FluentAssertions;
using NSubstitute;
using Xunit;

public class MyServiceTests
{
    [Theory, AutoNSubstituteData]
    public void DoWork_CallsDependency(
        [Frozen] IMyDependency dependency,  // NSubstitute substitute, injected into sut
        MyService sut,                      // constructed with dependency already wired
        string input)                       // random string from AutoFixture
    {
        sut.DoWork(input);

        dependency.Received(1).Process(input);
    }
}
```

> **Tip:** Parameters are resolved left to right. Place `[Frozen]` parameters before any types that depend on them.

## FixtureFactory

`FixtureFactory` creates `IFixture` instances pre-configured with NSubstitute auto-substitution. Use it in `[Fact]` tests where you need a fixture directly.

```csharp
// Plain fixture — AutoNSubstituteCustomization applied automatically
IFixture fixture = FixtureFactory.Create();

// With extra customizations
IFixture fixture = FixtureFactory.Create(
    new RecursionCustomization(),
    new ImmutableCollectionCustomization());
```

**Example `[Fact]` test:**

```csharp
[Fact]
public void ProcessOrder_SavesOrder_ToRepository()
{
    var fixture = FixtureFactory.Create();
    var repository = fixture.Freeze<IOrderRepository>();
    var sut = fixture.Create<OrderService>();
    var order = fixture.Create<Order>();

    sut.ProcessOrder(order);

    repository.Received(1).Save(order);
}
```

## Theory Data Attributes

All four attributes share the same fixture pipeline: `AutoNSubstituteCustomization` → project-wide `SutFixtureCustomizations` → `[CustomizeWith]` on the method → `[CustomizeWith]` on the class.

### `[AutoNSubstituteData]`

All theory parameters are auto-generated. Interfaces and abstract classes become NSubstitute substitutes.

```csharp
[Theory, AutoNSubstituteData]
public void Test(IMyService service, string value, int count)
{
    // service is an NSubstitute substitute
    // value and count are random AutoFixture values
}
```

### `[InlineAutoNSubstituteData]`

Explicit inline values fill the leading parameters; remaining parameters are auto-generated.

```csharp
[Theory]
[InlineAutoNSubstituteData("hello", 42)]
public void Test(string message, int count, IMyService service)
{
    // message == "hello", count == 42
    // service is an NSubstitute substitute
}
```

### `[MemberAutoNSubstituteData]`

Rows from a static member (property, field, or method) fill the leading parameters; remaining parameters are auto-generated. Each row gets its own fresh fixture.

```csharp
public static IEnumerable<object[]> InvalidInputs =>
[
    new object[] { null },
    new object[] { "" },
];

[Theory]
[MemberAutoNSubstituteData(nameof(InvalidInputs))]
public void Test(string? input, IMyService service)
{
    // input is null or ""
    // service is an NSubstitute substitute
}
```

Set `MemberType` to look up the member on a different class:

```csharp
[MemberAutoNSubstituteData(nameof(SharedData.Inputs), MemberType = typeof(SharedData))]
```

### `[ClassAutoNSubstituteData]`

Rows from a data class (implementing `IEnumerable<object[]>`) fill the leading parameters; remaining parameters are auto-generated.

```csharp
public class InvalidInputData : IEnumerable<object[]>
{
    public IEnumerator<object[]> GetEnumerator()
    {
        yield return new object[] { null };
        yield return new object[] { "" };
    }
    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}

[Theory]
[ClassAutoNSubstituteData(typeof(InvalidInputData))]
public void Test(string? input, IMyService service)
{
    // input is null or ""
    // service is an NSubstitute substitute
}
```

## Customizations

### `AutoNSubstituteCustomization`

Always applied by `FixtureFactory` and all theory data attributes. Configures AutoFixture to use NSubstitute for any interface or abstract class it cannot otherwise construct.

```csharp
new AutoFixture.AutoNSubstitute.AutoNSubstituteCustomization
{
    ConfigureMembers = true,
    GenerateDelegates = true,
}
```

You do not need to apply this manually — it is the foundation of every fixture created by this library.

### `RecursionCustomization`

Prevents AutoFixture from throwing `ObjectCreationException` on self-referencing or recursive object graphs. Replaces the default `ThrowingRecursionBehavior` with `OmitOnRecursionBehavior` (recursive references are left `null`).

```csharp
var fixture = FixtureFactory.Create(new RecursionCustomization());
var node = fixture.Create<TreeNode>(); // no exception even if TreeNode has Children: List<TreeNode>
```

### `ImmutableCollectionCustomization`

Enables AutoFixture to create instances of all eight immutable collection types from `System.Collections.Immutable`:

`ImmutableArray<T>` · `ImmutableList<T>` · `ImmutableDictionary<TKey, TValue>` · `ImmutableHashSet<T>` · `ImmutableSortedSet<T>` · `ImmutableSortedDictionary<TKey, TValue>` · `ImmutableQueue<T>` · `ImmutableStack<T>`

```csharp
var fixture = FixtureFactory.Create(new ImmutableCollectionCustomization());
var items = fixture.Create<ImmutableList<string>>();
```

## Project-Wide Customizations (`SutFixtureCustomizations`)

Register customizations once in your test assembly using `[ModuleInitializer]`. Every fixture created by any theory data attribute in the assembly will have these customizations applied automatically — before any `[CustomizeWith]` overrides.

```csharp
internal static class TestAssemblyInitializer
{
    [ModuleInitializer]
    public static void Initialize()
        => SutFixtureCustomizations.Add(new MyDomainCustomization());
}
```

`SutFixtureCustomizations` is append-only and thread-safe. There is no `Clear()` — use `[CustomizeWith]` for per-test overrides.

---

## Method/Class Customization (`[CustomizeWith]`)

Apply an `ICustomization` to a single test method or an entire test class without registering it project-wide.

```csharp
// Applied to every test in the class
[CustomizeWith(typeof(MyProjectCustomization))]
public class MyServiceTests
{
    // Applied only to this test (in addition to the class-level customization)
    [Theory, AutoNSubstituteData]
    [CustomizeWith(typeof(MySpecialCustomization))]
    public void Test(IMyService service) { ... }
}
```

The customization type must implement `ICustomization` and have a public parameterless constructor. Multiple attributes can be stacked; they are applied in declaration order.

## Frozen Parameters (`[Frozen]`)

`[Frozen]` (from AutoFixture) freezes a parameter in the fixture so that all subsequent parameters resolving the same type receive the same instance. This is the primary mechanism for injecting a known substitute into a SUT.

```csharp
[Theory, AutoNSubstituteData]
public void UpdateUser_PublishesEvent(
    [Frozen] IEventBus eventBus,    // frozen — same instance injected into sut
    [Frozen] IUserRepository repo,  // frozen — same instance injected into sut
    UserService sut,                // constructed with both frozen dependencies
    User user)
{
    sut.UpdateUser(user);

    repo.Received(1).Save(user);
    eventBus.Received(1).Publish(Arg.Any<UserUpdatedEvent>());
}
```

`[Frozen]` works with all four theory data attributes.
